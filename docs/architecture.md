# Architecture

## Purpose

This repository defines a custom Omarchy theme named `flat-onedark`.

Its job is to:

- define a stable One Dark visual system
- generate the repo-owned theme artifacts from one token file
- let Omarchy generate template-driven surfaces from `colors.toml`
- keep Waybar stable even when the Omarchy source tree is locally dirty

Its job is not to:

- own Omarchy itself
- edit `~/.local/share/omarchy`
- own qbar setup
- own the live Waybar layout beyond the repaired local stock base

## Design Principles

### 1. Single source of truth

`theme.tokens.sh` is the source of truth for:

- palette
- typography
- spacing and geometry tokens
- shared semantic values for generated outputs

Generated files should not become independent design sources.

### 2. Hybrid ownership

The repo deliberately mixes two models:

- repo-owned generated files for surfaces where structure or behavior matters
- Omarchy template-driven files for surfaces where palette is the main concern

This is not accidental. Full hardcoding creates drift. Full templating removes needed control.

### 3. Repairable local runtime

Waybar is the exception that forced a dedicated repair model.

Historically, the local Omarchy source tree contained modified Waybar defaults. Because `theme-manager` and `omarchy-refresh-waybar` copy from that source tree, “reset to default” still produced a broken Waybar.

Current design:

- the theme never edits `~/.local/share/omarchy`
- `scripts/repair-waybar.sh` rebuilds a clean local stock base in `~/.config/waybar/themes/omarchy-default`
- the live Waybar symlinks point to that repaired local copy

## System Model

### Authoring layer

Files edited by maintainers:

- `theme.tokens.sh`
- `README.md`
- `docs/*`
- `scripts/*`
- selected static assets such as `icons.theme`, `vscode.json`, `preview.png`, and `backgrounds/*`

### Generated repo layer

Files regenerated by `scripts/build-theme.sh`:

- `colors.toml`
- `colors.fish`
- `fzf.fish`
- `waybar.css`
- `walker.css`
- `gtk.css`
- `hyprland.conf`
- `starship.toml`
- `qbar.css`
- `steam.css`
- `vencord.theme.css`
- `neovim.lua`
- `cava_theme`
- `lazygit.yml`

These files are versioned, but they are not hand-edited sources.

### Omarchy template layer

Omarchy turns `colors.toml` into runtime files such as:

- `alacritty.toml`
- `kitty.conf`
- `ghostty.conf`
- `hyprlock.conf`
- `mako.ini`
- `swayosd.css`
- `btop.theme`
- `chromium.theme`
- `obsidian.css`
- `keyboard.rgb`

Those files should not exist in the repo root unless this theme is intentionally taking ownership away from Omarchy templates.

### Local runtime layer

At runtime the active theme lives in:

- `~/.config/omarchy/current/theme`

Waybar live state lives in:

- `~/.config/waybar/config.jsonc`
- `~/.config/waybar/style.css`
- `~/.config/waybar/themes/omarchy-default/`

The crucial point is that the repaired local Waybar base is part of runtime state, not part of the Omarchy source tree.

## Waybar Special Case

Waybar is special for two reasons:

1. it mixes Omarchy stock structure with user-local live files
2. the upstream local Omarchy source tree on this machine is dirty for `config/waybar/*`

That created the old bug pattern:

- theme or theme-manager “reset” appears to succeed
- Waybar still looks wrong
- the user assumes the current theme is broken
- the real problem is that “default” itself is contaminated

Current contract avoids this:

- the theme owns only `waybar.css`
- the repair command owns the local stock base in `~/.config/waybar/themes/omarchy-default`
- qbar must own its own modules and icons if enabled

## qbar Policy

`qbar.css` exists only as a visual contract for future integration.

This repository does not:

- install qbar
- add qbar modules to the repaired stock Waybar base
- inject qbar CSS into the live default bar

If qbar is wanted, `qbar setup` must own that integration.

## Documentation Rule

Operational truth belongs in `docs/`.

Historical decision records belong in `docs/plans/`.

Do not treat a planning file as the current implementation contract unless it has been folded into the operational docs.
